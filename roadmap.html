Logika Peran Akses (RBAC)

SELESAI

Akses antara Admin dan User biasa berhasil dipisahkan dengan menyuntikkan Role ke dalam JWT klien.

  • Mekanisme Keamanan: Role telah berhasil diikat ke dalam raw_app_meta_data Supabase Auth dan dikawal oleh router.js.
  • Contoh perintah SQL Editor untuk menetapkan pengguna pertama sebagai Super Admin:
-- Ganti email dengan email akun Anda yang terdaftar
UPDATE auth.users 
SET raw_app_meta_data = raw_app_meta_data || '{"role": "admin"}' 
WHERE email = 'admin@example.com';

Keamanan Supabase (RLS)

Menerapkan Row Level Security pada Database agar API publik tidak bisa memanipulasi data tanpa izin, divalidasi lewat JWT Role.

Aturan Spesifik RLS (Contoh untuk Tabel):

  • SELECT (Akses Baca): Semua pengguna yang Authenticated (baik user maupun admin) berhak membaca daftar.
  • INSERT/UPDATE/DELETE (Akses Tulis): HANYA pemilik JWT yang memiliki atribut metadata role = 'admin' yang berhak mengubah data base.
-- Kebijakan SELECT
CREATE POLICY "Bolehkan baca untuk authenticated users" ON nama_tabel
FOR SELECT USING (auth.role() = 'authenticated');

-- Kebijakan TULIS/MODIFIKASI (Hanya Admin)
CREATE POLICY "Hanya Admin yang boleh Tulis" ON nama_tabel 
FOR ALL USING (auth.jwt() -> 'app_metadata' ->> 'role' = 'admin');

Siklus Sesi & Pembaruan Token (Token Refresh)

Manajemen daur hidup sesi login (Lifecycle) agar SPA kita kebal dari error saat JWT kedaluwarsa.

  • Penyegaran Otomatis: Supabase JS Client (di auth.js) sebenarnya sudah menangani penyegaran token (refresh token) secara berkala di latar belakang.
  • Penanganan Sesi Mati: Kita akan menanamkan fungsi supabase.auth.onAuthStateChange() secara global. Jika token benar-benar mati (expired tanpa bisa direfresh) atau di-revoke oleh server, fungsi ini akan mencegat event SIGNED_OUT dan seketika menendang pengguna ke #login.
  • Mencegah Crash UI: Ini memastikan aplikasi tidak mencoba menarik data (Fetch Data) dengan token hangus yang akan berujung pada HTTP 401 dan layar blank/tampilan loading abadi.
